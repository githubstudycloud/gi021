# 上下文管理：压缩、清理与成本控制

> 上下文是 Claude Code 的生命线。70% 时开始降精度，85% 时幻觉增加，90%+ 时响应混乱。

---

## 一、上下文窗口原理

Claude Code 使用 200K token 的上下文窗口，但 **有效工作区间远小于此**：

| 上下文使用率 | 效果 |
|-------------|------|
| < 70% | 正常，精度最高 |
| 70%–85% | 开始降精度，忽略末尾内容 |
| 85%–90% | 幻觉增加，容易编造不存在的 API |
| > 90% | 响应混乱，无法可靠完成任务 |

**什么会消耗上下文？**
- 对话历史（每轮问答都会累积）
- 读取的文件内容
- 工具调用结果（特别是长日志、大文件）
- MCP 服务器注入的内容（可能消耗 10-20k token）

---

## 二、三个核心命令

### `/clear` — 清空上下文

```
> /clear
```

清空整个对话历史，回到"白板"状态。**CLAUDE.md 的内容不受影响**（每次对话开始都会重新加载）。

**何时使用**：
- 切换到完全不相关的任务时
- 上下文已严重污染（错误的思路、过期的假设）
- 任务完成，开始新任务

### `/compact` — 压缩上下文

```
> /compact
> /compact 保留认证模块的讨论，其他可以摘要化
```

把旧的对话历史压缩为摘要，释放 token 空间，同时保留关键信息。适合**长会话中途**使用。

**何时使用**：
- 单个任务持续很长时间（探索了很多文件）
- 想继续当前任务但上下文已接近限制
- 需要保留部分历史但可以省略细节

### `/resume` — 恢复历史会话

```
> /resume
```

弹出历史会话选择器，可以恢复之前的对话（含完整历史）。

## 三、会话恢复

### 命令行启动时恢复

```bash
# 自动继续最近一次会话
claude --continue

# 弹出历史会话选择器
claude --resume

# 在非交互模式下继续（脚本友好）
claude --continue --print "继续之前的任务，上次我们完成了登录功能"
```

### 会话存储位置

对话历史保存在本地（不同平台路径不同），**不会上传到云端**：
- macOS/Linux：`~/.claude/` 目录下
- Windows：`%USERPROFILE%\.claude\` 目录下

## 四、主动管理上下文

### 策略 1：任务切换时清理

```
# 完成任务 A，开始任务 B 前
> /clear

# 重新加载必要上下文
> 我现在要处理登录页面的问题，相关文件在 @src/pages/Login.tsx
```

### 策略 2：使用 `@` 引用精准加载

不要让 Claude 漫无目的地探索，精确告知需要哪些文件：

```
请参考 @src/components/UserProfile.tsx 的结构实现新组件
对比 @docs/api.md 和 @src/api/users.ts 检查接口是否一致
```

避免引用整个大目录（会加载大量文件）：
```
# ❌ 可能加载太多内容
分析 @src/ 目录

# ✅ 精准引用
分析 @src/auth/login.ts 和 @src/auth/types.ts
```

### 策略 3：用 `#` 把重要信息写入 CLAUDE.md

不要依赖 Claude 在长会话中记住事情，把关键约定写入永久记忆：

```
# 记住：支付模块使用 Stripe SDK v4，不是 v3，接口有破坏性变更
# 记住：测试数据库 URL 在 .env.test 中，不要用生产环境数据库
```

Claude 会把这些信息追加到当前项目的 CLAUDE.md 中，下次会话自动加载。

### 策略 4：定期检查上下文状态

```
> /status
```

查看当前 token 使用量，在上下文超过 70% 之前主动压缩或清理。

## 五、减少 Token 消耗

### 控制工具输出量

```bash
# 在 ~/.claude/settings.json 中设置
{
  "env": {
    "CLAUDE_CODE_MAX_OUTPUT_TOKENS": "128000"
  }
}
```

### 禁用非必要模型调用

```json
{
  "env": {
    "DISABLE_NON_ESSENTIAL_MODEL_CALLS": "1"
  }
}
```

### 禁用 Prompt Caching（节省成本但可能略慢）

```json
{
  "env": {
    "DISABLE_PROMPT_CACHING": "1"
  }
}
```

### MCP 服务器 Token 控制

> MCP 服务器会将内容注入上下文，>20k token 的 MCP 配置会严重影响质量。

- 只安装当前任务需要的 MCP
- 任务完成后考虑禁用不用的 MCP
- 避免配置多个重量级 MCP 同时运行

## 六、长任务的上下文策略

### 分段执行模式

将大任务分解为多个独立的小会话：

```bash
# 会话 1：探索 + 计划
claude
> 请分析认证模块，制定重构计划，不要写代码
> /compact  # 保留计划，压缩探索过程

# 会话 2：执行第一部分
claude
> 请执行认证重构计划的第 1-3 步

# 会话 3：执行第二部分
claude --continue
> 继续执行第 4-6 步
```

### SCRATCHPAD 模式

让 Claude 将计划、进度写入文件，而不是保存在对话历史中：

```
请将实现计划写入 docs/plan.md，然后我们开始执行。
每完成一个步骤，更新 docs/plan.md 中的进度。
```

这样即使清空上下文，计划仍在文件中。

## 七、成本监控

### 查看使用情况

```bash
> /status   # 查看当前会话 token 使用量
```

### 估算成本（2026 年参考价格）

| 模型 | 输入 | 输出 |
|------|------|------|
| Opus 4.6 | $15/M tokens | $75/M tokens |
| Sonnet 4.6 | $3/M tokens | $15/M tokens |
| Haiku 4.5 | $0.8/M tokens | $4/M tokens |

> 日常开发推荐 Sonnet 4.6，复杂架构决策用 Opus，简单任务用 Haiku。

### 降低成本的技巧

1. **使用 `/compact` 而非 `/clear`** — 避免重复加载相同文件
2. **精准 `@` 引用** — 只加载真正需要的文件
3. **`-p` 模式用于批处理** — 不进入交互模式，减少来回消耗
4. **合理使用扩展思考** — `think` 触发额外 token 消耗，仅在复杂任务使用

## 八、最佳实践总结

```
会话开始   → /init 或 /status 检查环境
探索阶段   → 精准 @引用，避免大范围扫描
计划阶段   → ultrathink + 写入文件持久化
执行阶段   → 定期检查上下文使用率
> 70%      → /compact（保留关键信息）
任务切换   → /clear（重置上下文）
重要约定   → # <内容> 写入 CLAUDE.md
跨天任务   → claude --resume 恢复历史会话
```
